# دليل العمل للوكيل الذكي

هذا الملف هو handoff مختصر لأي وكيل AI جديد سيعمل على مشروع `MohamyPhone`.
اقرأه أولًا قبل أي تعديل، ثم نفّذ التغييرات مباشرة داخل الريبو.

## 1) ما هو المشروع

`MohamyPhone` تطبيق Android مخصص لإدارة مكتب محاماة عربي.

الهدف:
- إدارة الموكلين
- إدارة القضايا
- إدارة الجلسات
- إدارة المهام
- إدارة الملفات والمستندات
- توليد قوالب ومستندات قانونية
- مساعد ذكي قانوني داخل التطبيق
- صفحة Admin على الويب لإدارة التراخيص والعملاء
- نظام تحديث من GitHub Releases

التطبيق يعتمد على `local-first`:
- البيانات الأساسية محفوظة محليًا على الجهاز
- التفعيل والترخيص عبر السيرفر فقط
- التحديثات عبر GitHub Releases

## 2) التقنيات الأساسية

- Android Kotlin
- Jetpack Compose
- Room Database
- ViewModel + Coroutines
- FileProvider وفتح الملفات محليًا
- GitHub Actions للتحديثات
- صفحة Admin على الويب داخل نفس الريبو

## 3) اللغة والأسلوب

- واجهة المستخدم عربية
- التسميات القانونية في الواجهة يجب أن تكون احترافية
- استخدم مصطلحات مثل:
  - `الموكلون`
  - `القضايا`
  - `الجلسات`
  - `المساعد الذكي`
  - `المكتبة`
- لا تغيّر الفكرة العامة للتطبيق إلى SaaS سحابي كامل
- لا تحذف منطق التخزين المحلي إلا إذا كان هناك سبب قوي

## 4) ما الذي يجب أن يظل ثابتًا

- `applicationId` الحالي
- نظام التحديث عبر GitHub Releases
- نظام التفعيل الحالي
- البيانات المحلية لكل حساب على نفس الجهاز
- التوافق مع النسخة المثبتة عند المستخدمين

## 6) طريقة الشغل المطلوبة

عند طلب أي تعديل:
1. افهم الهدف العملي أولًا.
2. عدّل الكود مباشرة.
3. اربط التعديل بباقي الشاشات أو الداتا لو لازم.
4. اختبر البناء محليًا.
5. لو التعديل يؤثر على السلوك المرئي، شغّل التطبيق على المحاكي وتأكد من النتيجة.
6. لو التعديل يخص التحديث، حدّث أيضًا `update/latest.json` و`versionCode/versionName`.

## 7) أسلوب التنفيذ المفضل

- استخدم `rg` للبحث
- استخدم `apply_patch` للتعديلات الصغيرة والمتوسطة
- لا تكتب تغييرات تجميلية فقط إذا كانت المشكلة سلوكية
- لا تلمس ملفات لا علاقة لها بالمهمة
- لا تعكس تغييرات المستخدم أو تحذفها بدون إذن
- لا تعتمد على الرد النظري؛ نفّذ واحقق ثم سلّم

## 8) اختبارات أساسية بعد أي تعديل مهم

### على Android
- `./gradlew :app:assembleDebug`
- `./gradlew :app:testDebugUnitTest`
- `./gradlew :app:assembleRelease`

### على المحاكي
- ثبّت `app-release.apk` أو `app-debug.apk`
- شغّل `com.example.MainActivity`
- تأكد من:
  - الإقلاع
  - التنقل بين الصفحات
  - المساعد الذكي
  - القضايا
  - الملفات
  - الجلسات
  - التحديث

## 9) قواعد مهمة جدًا في هذا الريبو

- يوجد في الشجرة غالبًا ملفات محذوفة قديمة من تقارير أو صور QA
- لا تسترجع هذه الملفات ولا تعتبرها جزءًا من المهمة
- توجد مجلدات/ملفات محلية غير مهمة مثل:
  - `signing/`
  - `tmp-emulator-home.png`
- لا تدخلها في commit إلا إذا كانت مطلوبة فعلًا

## 10) نظام التحديث

النسخة النهائية تُنشر عبر:
- GitHub Release
- `update/latest.json`
- workflow: `.github/workflows/publish-release.yml`

القاعدة:
- زِد `versionCode` في كل إصدار جديد
- حدّث `versionName`
- حدّث `update/latest.json`
- أنشئ tag جديد مثل `v1.3.1`

لو رفعت نسخة جديدة ولم تحدّث `versionCode`:
- التحديث قد يفشل في التثبيت

## 11) فلسفة المنتج

التطبيق يجب أن يبدو كمنتج مهني حقيقي، وليس نموذجًا تجريبيًا.

أولوياتنا:
- وضوح
- سرعة
- سلاسة
- بيانات مرتبة
- لغة قانونية محترمة
- واجهة عربية مقروءة
- ربط جيد بين الشاشات

## 12) مساعد ذكي

المساعد الذكي ليس مجرد أزرار.

المفروض:
- يعمل كشات بوت
- يفهم القضية الحالية
- يعرض رسائل واضحة عند عدم وجود بيانات
- يربط بين:
  - القضايا
  - الجلسات
  - الملفات
  - المهام
  - القوالب

لو طلبت من المساعد شيء مثل:
- ملخص القضية
- تجهيز الجلسة القادمة
- المستندات الناقصة
- آخر جلسة
- مسودة مذكرة

يجب أن:
- يرجع ردًا مهنيًا
- أو يوضح أن البيانات غير كافية

## 13) الملفات والقوالب

التطبيق يحتوي على:
- ملفات القضية
- مكتبة الملفات
- قوالب قانونية
- مستندات مولدة

المبدأ:
- كل ملف له هوية مرئية قابلة للتخصيص
- القوالب يجب أن تكون قانونية ومهنية
- لو أضفت قالبًا أو وثيقة جديدة، تأكد أنها قابلة للحفظ والعرض والتصدير

## 14) لو أردت أن تبدأ مهمة جديدة

استخدم هذا النمط:

> اعمل [الهدف]
> ركز على [الصفحات/الملفات]
> اختبر على المحاكي
> حدّث GitHub لو لزم
> لا تغيّر [الجزء الذي يجب تركه]

## 15) ملاحظات عملية

- لو وجدت تعارضًا بين الجودة والسرعة: حسّن البنية أولًا
- لو وجدت شاشة ضعيفة بصريًا: أعد ترتيبها بدل إضافة عناصر كثيرة
- لو وجدت خطأ في البيانات: عالج مصدر البيانات وليس العرض فقط
- لو وجدت أن المستخدم لا يفهم شيئًا: أضف empty state ورسالة خطأ واضحة

## 16) ماذا ترد لو احتجت توضيحًا

إذا كان هناك غموض حقيقي لا يمكن استنتاجه من الكود:
- اسأل سؤالًا واحدًا مباشرًا
- لا تكثر أسئلة
- لا توقف التنفيذ إلا للضرورة
